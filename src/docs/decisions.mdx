# Design Decisions

**Why this exists:** Recorded trade-offs — what we chose, why, and what we gave up. Not a process doc; a memory aid for contributors.

## Local-first architecture

**Decision:** IndexedDB for all training data. No backend required.

**Why:** Fast access, offline training, privacy, simpler deployment.

**Trade-off:** No sync or multi-device backup today.

## First-move verification

**Decision:** Compare the user's move to the expected **first move** (UCI). Full solution lines are not verified interactively.

**Why:** Emphasizes pattern recognition and faster feedback — aligned with Woodpecker-style training.

**Trade-off:** Does not prove full-line calculation.

## Cycle-based organization

**Decision:** Training organized as repeated **cycles** through fixed sets.

**Why:** Core to the method; enables time-reduction tracking and pattern memory.

**Trade-off:** Less variety than random puzzle feeds.

## Training vs reflection (routes)

**Decision:** `/app/training` stays in execution mode. No active cycle → empty state, not auto-redirect to last summary.

**Why:** Separates doing from reviewing; avoids surprise navigation.

**Reflection charts:** Scoped **per training set** — cycle number vs time for one set only.

## Mistake review

**Decision:** Failed positions enter a review queue with mastery progression.

**Why:** Turns failures into targeted practice.

**Trade-off:** Additional state to maintain.

## No authentication

**Decision:** No user accounts.

**Why:** Simpler architecture; data stays local.

**Trade-off:** No sync, sharing, or cloud backup.

## TypeScript throughout

**Decision:** Strict TypeScript for all application code.

**Why:** Catch errors early; schemas self-document persistence boundaries.

**Trade-off:** Slightly more ceremony for contributors.
