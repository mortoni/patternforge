# Local-First Architecture

**Why this exists:** Core training should work without a backend, survive offline use, and keep user data on the device.

## What local-first means here

- **IndexedDB** stores sets, cycles, sessions, attempts, mistakes, and settings
- **No account required** for training flows
- **Fast reads/writes** — no network round-trip for progress
- **Privacy by default** — data stays in the browser (same origin)

Static puzzle bundles in `public/data/woodpecker/` are fetched when the app loads, then upserted into Dexie. After seeding, training runs against local data.

## Engineering implications

- Repositories in `src/repositories/` are the persistence boundary
- Zod schemas in `src/db/schema.ts` validate rows at write time
- Services orchestrate multi-table updates (cycle completion, session close)
- Tests use Vitest with jsdom; Dexie can be exercised in unit tests
